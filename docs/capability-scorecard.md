# Capability Scorecard

**Status: In progress — template.** This is a reusable, living tracker of skill development
across all projects in the lab, not a one-time assessment. Update it as projects produce
evidence; do not backfill confidence scores without evidence to support them.

## How to use this table

- Only update rows for capabilities that were materially exercised during the current stage. A
  row with no new evidence this stage should simply be left as-is, not padded to look current.
- **Current evidence** — a link to a dated, concrete repository artifact: a doc, a test, a trace,
  a dataset, or a decision log entry. "None yet" is a valid, honest entry. Self-assessment without
  a linked artifact is not evidence.
- **Gaps** — what's missing or weak, stated plainly.
- **Next exercise** — the smallest next action that would produce new evidence.
- **Confidence (1–5)** — grounded in the linked evidence, not self-assessment alone. 1 = no real
  evidence yet. 5 = repeatedly demonstrated across more than one project, with evidence for each.
  Do not round up. "Debugging" confidence in particular should reflect demonstrated ability to
  explain execution paths and narrow failures (see Technical ownership in
  `docs/lab-operating-standard.md`), not the ability to get an issue resolved by any means.

## Notes

This scorecard is intentionally coarse. It is meant to reveal where the lab's practice is thin,
not to produce a precise skill measurement. Revisit and update it at the end of each project
stage (Reality, Build, Break, Repair, Abstract), not just at project completion.
